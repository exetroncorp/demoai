00

Hello World

The simplest possible call to the Responses API. Replace $OPENAI_API_KEY with your key or export it as an environment variable.

bash · minimal hello world 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "input": "Hello! What is the Responses API?"
  }'
💡
Tip

The key difference from Chat Completions: instead of a messages array, you pass a single input string (or array). The API is stateful by default — responses are stored server-side and can be continued.

You'll get back a JSON object with your answer inside output[0].content[0].text. Let's break down every part of the request and response next.

01

Request Body Anatomy

Here's a full-featured request showing all the important fields:

bash · full request body 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model":       "gpt-4o",
    "input":       "Explain quantum entanglement simply.",
    "instructions":"You are a physics teacher for curious 10-year-olds.",
    "temperature": 0.7,
    "max_output_tokens": 300,
    "store":       true
  }'

Field reference:

Field Type Status Description
model string required Model ID, e.g. gpt-4o, gpt-4o-mini, o1
input string | array required Single string OR array of message objects with role/content
instructions string optional System-level instructions (replaces the old system role message)
temperature float optional Randomness 0–2. Default: 1. Lower = more deterministic
max_output_tokens int optional Maximum tokens in the response
store bool optional Whether to store the response server-side. Default: true
previous_response_id string optional ID of a previous response to continue a conversation
tools array optional Built-in tools like web_search, file_search, code_interpreter
stream bool optional Enable server-sent events streaming
02

Roles & Input Types

When input is an array, each item is a message object with a role. The Responses API supports three roles:

RoleWhoPurpose
user End user / human What the human says or asks
assistant The model Previous AI responses (for context)
system Developer Behaviour instructions (prefer instructions field)
bash · input as array with roles 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "input": [
      {
        "role": "user",
        "content": "What is 15 multiplied by 37?"
      }
    ]
  }'

The content field inside a message can itself be a string or an array of typed content parts (text, image_url, etc.):

bash · multimodal content parts 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "input": [
      {
        "role": "user",
        "content": [
          {
            "type": "input_text",
            "text": "What is in this image?"
          },
          {
            "type": "input_image",
            "image_url": "https://example.com/photo.jpg"
          }
        ]
      }
    ]
  }'
03

Response JSON

A typical response looks like this:

json · response object 
{
  "id": "resp_abc123xyz",
  "object": "response",
  "created_at": 1714000000,
  "model": "gpt-4o-2024-11-20",
  "status": "completed",
  "output": [
    {
      "type": "message",
      "role": "assistant",
      "content": [
        {
          "type": "output_text",
          "text": "Hello! The Responses API is OpenAI's..."
        }
      ],
      "id": "msg_def456",
      "status": "completed"
    }
  ],
  "usage": {
    "input_tokens": 12,
    "output_tokens": 94,
    "total_tokens": 106
  },
  "incomplete_details": null,
  "error": null
}

Key fields explained:

id
Unique response ID. Save this to continue the conversation with previous_response_id.e.g. "resp_abc123xyz"
status
Lifecycle state of the response."completed" | "in_progress" | "failed" | "incomplete"
output
Array of output items (messages, tool calls, tool results…). The AI text lives at output[0].content[0].text.type: "message" | "function_call" | "web_search_call" …
usage
Token counts for billing and context management.input_tokens + output_tokens = total_tokens
error
Null on success. Contains code + message on failure.null | { code, message }
04

System Prompt / Instructions

In the Responses API, the recommended way to set system-level behaviour is the top-level instructions field — cleaner than injecting a system role message.

bash · instructions field 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model":        "gpt-4o",
    "instructions": "You are a sarcastic but helpful pirate assistant. Always end with Arrr!",
    "input":        "What is the capital of France?"
  }'
⚠️
Note

instructions apply to every turn in a stateful session. If you use previous_response_id to continue, the instructions from the latest call take effect.

05

Multi-turn Conversations

The Responses API stores history server-side. You don't need to replay the whole conversation — just pass previous_response_id.

bash · turn 1: ask a question 
# TURN 1 — start the conversation
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "input": "My name is Robert and I love Docker.",
    "store": true
  }'

# ↑ Response will contain: { "id": "resp_TURN1ID", ... }
bash · turn 2: continue with previous_response_id 
# TURN 2 — reference the previous response
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "previous_response_id": "resp_TURN1ID",
    "input": "What is my name and what do I love?"
  }'

# Model knows: "Your name is Robert and you love Docker!"
06

Streaming SSE

Add "stream": true to receive server-sent events as the model generates tokens. The curl flag --no-buffer ensures output is printed in real-time.

bash · streaming response 
curl https://api.openai.com/v1/responses \
  --no-buffer \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model":  "gpt-4o",
    "input":  "Write a short poem about containers.",
    "stream": true
  }'

You'll see a stream of events like this in your terminal:

sse · stream output 
event: response.created
data: {"type":"response.created","response":{"id":"resp_xyz","status":"in_progress",...}}

event: response.output_item.added
data: {"type":"response.output_item.added","output_index":0,"item":{...}}

event: response.output_text.delta
data: {"type":"response.output_text.delta","delta":"In "}

event: response.output_text.delta
data: {"type":"response.output_text.delta","delta":"containers "}

event: response.output_text.done
data: {"type":"response.output_text.done","text":"In containers cold and..."}

event: response.completed
data: {"type":"response.completed","response":{"status":"completed","usage":{...}}}
07

Built-in Tools

Unlike Chat Completions, the Responses API ships with hosted tools you can enable with a single line — no function definitions required.

bash · web search tool 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "tools": [{ "type": "web_search_preview" }],
    "input": "What are the latest Docker release notes in 2025?"
  }'
bash · code interpreter tool 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "tools": [{ "type": "code_interpreter", "container": { "type": "auto" } }],
    "input": "Calculate the first 10 Fibonacci numbers and plot them."
  }'
bash · multiple tools at once 
curl https://api.openai.com/v1/responses \
  -H "Authorization: Bearer $OPENAI_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "tools": [
      { "type": "web_search_preview" },
      { "type": "code_interpreter", "container": { "type": "auto" } }
    ],
    "input": "Find the latest Bitcoin price and graph the last 7 days."
  }'

Available built-in tools:

Tool typeWhat it does
web_search_previewLive web search with citations
file_searchSemantic search over uploaded files / vector stores
code_interpreterPython sandbox — runs code, generates files
functionCustom function (same as Chat Completions tool calling)
computer_use_previewControl a virtual desktop (beta)
08

Responses vs Chat Completions

Both APIs coexist. Here's a quick migration comparison:

Chat Completions (old)

"messages": [
  {"role": "system", "content": "..."},
  {"role": "user",   "content": "..."}
]
// No server-side state
// You manage history manually
// Tools via function definitions

Responses API (new)

"instructions": "...",
"input": "..."
// Server-side state (store: true)
// Continue via previous_response_id
// Built-in hosted tools
FeatureChat CompletionsResponses API
Stateful history❌ manual✅ server-side
Web search❌✅ built-in
Code interpreterAssistants only✅ built-in
Multimodal input✅✅
Streaming✅✅
System promptsystem roleinstructions field
Response fieldchoices[0].message.contentoutput[0].content[0].text
