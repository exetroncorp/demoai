🎛️Modes — The Core Paradigm

Modes are specialized AI personas that tailor Bob's behavior, tool access, and response style for specific tasks. Choosing the right mode is the single biggest lever for quality output.

💡 Golden Rule: Plan first → Code second. Always start complex work in Plan mode, then switch to Code mode to implement.

Built-in Modes

💻
Code Mode
Write, modify, and refactor code. The default day-to-day mode for most developers. Optimized for cost efficiency.
read edit command
❓
Ask Mode
Conversational Q&A about your code. Safe mode — Bob cannot modify files. Perfect for code exploration and learning.
read browser mcp
📝
Plan Mode
Architecture, design, and technical planning before implementation. Bob acts as a senior tech lead gathering context to write a detailed spec.
read edit (md only) browser mcp
🛠️
Advanced Mode
Power user mode. Unrestricted access to all tool groups including MCP servers and browser automation. No restrictions.
read edit command mcp
🔀
Orchestrator Mode
Breaks complex multi-domain projects into sub-tasks and delegates them to the appropriate specialized modes automatically. No direct tool access — it coordinates other modes.
no direct tools

How to Switch Modes

  • Dropdown: Click the mode selector to the left of the chat input.
  • Slash command: Type /plan, /code, /ask, /advanced, or /orchestrator at the start of your message.
  • Keyboard: ⌘. (macOS) or Ctrl. (Windows/Linux) cycles through modes.
  • Bob suggestion: Accept the mode-switch suggestion Bob offers when it detects a task type mismatch.

📚Context Window Management

The context window is the "memory" Bob has during a conversation. It holds the conversation history, file contents, tool results, and your custom rules. Bob's context window has a maximum capacity of 200,000 tokens. Managing it properly is critical for response quality and Bobcoin efficiency.

Fact: Bob auto-condenses context at 140,000 tokens — before the 200k hard limit. It preserves recent messages and summarizes older ones. You can monitor current token usage in the top-right corner of the chat panel in real time.

What counts toward your token budget

Context componentToken impactTip
Conversation historyGrows with every messageStart new chats for unrelated tasks
@file mentionsFull file content added each messageUse @file.java:45-67 line ranges
@/folder/ mentionsCan be very large — use sparinglyPrefer specific files over directories
Tool outputsCommand output, file reads accumulateAvoid overly verbose commands
MCP tool definitionsEach connected server adds overheadDisconnect unused MCP servers
Custom rules / SkillsLoaded per conversationKeep rules concise and relevant
  • Start new conversations for new, unrelated tasks — don't reuse long threads.
  • Use @file mentions to add specific files rather than dumping your whole codebase.
  • Break large tasks into smaller focused sub-tasks.
  • Watch for "context poisoning" — when earlier errors in the conversation confuse later responses.
  • When Bob starts producing degraded output, restore a checkpoint and start fresh rather than trying to fix it with more prompts.
⚠️ Context Poisoning: If Bob makes a wrong assumption early in a long thread, it can "poison" all subsequent responses. Symptom: Bob keeps repeating the same mistake despite corrections. Fix: restore a checkpoint or start a new conversation.

💾Checkpoints

Checkpoints are automatic snapshots of your workspace files taken before Bob makes changes. They are your safety net — think of them as lightweight local git commits managed by Bob.

How Checkpoints Work

  • Bob automatically creates a checkpoint before every batch of file changes.
  • Checkpoints appear as markers in your chat history timeline.
  • You can scroll up in chat and click Restore Checkpoint at any point.
  • Two restore options: Restore Files only (keep conversation) or Restore Files & Task (full rollback).
💡 Best use: If Bob starts producing poor output after several turns, use Restore Files & Task to roll back completely rather than trying to correct it with more prompts. A fresh context beats a corrective prompt almost always.

⚙️Code Actions

Code Actions are contextual AI suggestions that appear directly in the editor — similar to VS Code's native quick-fix lightbulb, but powered by Bob.

Setting up IDE Integration (3 methods)

Method 1 — Automatic (recommended): run Bob Shell inside Bob IDE's terminal
Open the integrated terminal inside Bob IDE and run bob-shell. Bob Shell detects it's inside Bob IDE and shows an automatic prompt:

"Do you want to connect Bob IDE to Bob Shell?"
Select Yes. The Bob Shell Companion extension installs automatically and the connection is established. Done.
Method 2 — Manual install command
If you dismissed the auto-prompt, run this slash command inside Bob Shell:
/ide install
Bob detects your IDE, finds the Companion extension, installs it, and configures the connection.
Method 3 — Docker container
Bob Shell in a Docker container can still connect to Bob IDE on the host via host.docker.internal. Ensure container-to-host networking is allowed. The permissive-open Seatbelt profile (macOS) supports this.

🌟Best Practices

🔄 Workflow Strategy

  • Plan → Code → Test: Always start new features in Plan mode. Switch to Code only after reviewing the plan.
  • Small, focused tasks: One task per conversation. Don't chain unrelated requests.
  • Commit frequently: Treat Bob like a junior dev pair — commit after every logical unit of work.
  • Use Orchestrator for multi-domain work: If a task touches both backend and frontend in complex ways, Orchestrator mode coordinates better than doing it manually.

🔒 Security & Control

  • Keep .bobignore up to date — especially for .env, secrets, and PII-containing files.
  • Review Bob's proposed commands carefully before approving — especially rm, curl, or git push.
  • Use mode-specific rules to restrict what Bob can do in sensitive project contexts.
  • Prefer Manual approval mode in production-adjacent projects.

💰 Bobcoin Efficiency

  • Use Ask mode for exploration — it's cheaper than Code mode and doesn't need edit access.
  • Use Code mode (not Advanced) for standard tasks — Advanced mode is more expensive.
  • Keep context focused. Large contexts = more tokens = more Bobcoins consumed.
  • Set per-user Bobcoin budgets in the Enterprise dashboard to prevent runaway consumption.

✍️Effective Prompting

The quality of your output is directly proportional to the quality of your prompt. Here's what the IBM Bob team has found works best.

✅ Good prompts

Create a React component that displays a sortable table of user data.
The table should support sorting by name, email, and creation date.
Use TypeScript, Tailwind CSS, and shadcn/ui. Add aria labels for accessibility.

❌ Vague prompts

Make me a table component.

Prompting Principles

  • Be specific and clear: State what to do and what NOT to do.
  • Provide examples: "Format it like this: [example]" works better than "format it nicely".
  • Use Enhance Prompt: Click ✨ to let Bob improve your prompt automatically — great for complex or poorly worded requests.
  • Reference files with @: Point Bob at specific files instead of describing them.
  • State constraints: Budget, performance, framework version, team conventions.
  • One objective per message: Don't mix "refactor this function" with "also add a unit test" in one prompt if they're not related.

🔒Security Guidelines

Bob is a powerful tool. These guidelines help your team use it responsibly.

Data Handling

  • Prompts and code context are sent to IBM's servers for inference. Do not include secrets, PII, or regulated data in prompts.
  • Enterprise plans: IBM does not use your code or prompts for training. Confirm this with your IBM agreement.
  • Use .bobignore to exclude sensitive files from Bob's access.

Permission Model

  • Bob runs with the file system permissions of the logged-in user. It cannot escalate privileges.
  • MCP servers run with the credentials of whatever account you configured — scope them minimally.
  • Auto-approve for commands is the highest-risk setting. Require explicit confirmation for any command involving network, git push, or file deletion.
🚨 Critical: Never allow Bob to access production databases, live infrastructure credentials, or deployment keys directly. Always use dedicated dev/sandbox environments.

🧩Skills — What They Are

Skills are reusable, shareable prompt templates and workflow recipes that you can load into a Bob conversation to guide its behavior for a specific task type. Think of them as a saved "instruction booklet" you hand Bob before starting a complex job.

One-liner: A Skill = a pre-written set of instructions + context loaded into Bob's prompt. A Mode = a persistent AI persona with defined tool permissions that shapes ALL interactions in that session.

How Skills work in practice

  • A Skill is a Markdown (.md) file — usually named SKILL.md — containing domain knowledge, step-by-step procedures, constraints, and examples.
  • You activate a Skill by referencing it with @path/to/SKILL.md in your Bob chat message, or by placing it in your .bob/rules/ directory so it loads automatically.
  • Bob reads the Skill document and follows its instructions for the duration of that conversation or task.
  • Skills are portable — commit them to git and every developer on the team benefits from the same repeatable workflow.

Anatomy of a Skill file

# SKILL.md — Java Modernization Assistant
## Purpose
Guide Bob through upgrading a Java 8 codebase to Java 21 LTS.

## Prerequisites
- Java 21 target version
- Maven or Gradle build system
- Tests must pass before and after

## Step-by-step procedure
1. Run @pom.xml or @build.gradle analysis in Ask mode first.
2. Identify deprecated APIs using Bob's diagnostics view.
3. Switch to Code mode. Upgrade compiler source/target.
4. Update dependencies one by one. Verify build after each.
5. Replace java.util.Date with java.time.* throughout.
6. Replace raw types with generics where safe.
7. Run full test suite. Fix failures before proceeding.

## Output format
After each change, summarize: what changed, why, and any risks.

## Constraints
- NEVER change business logic. Modernize syntax only.
- ALWAYS keep a checkpoint before any batch of changes.
- DO NOT introduce new dependencies unless explicitly asked.

🚀Java Modernization with Bob

Modernizing legacy Java is Bob's marquee use case. Here's how to do it systematically, safely, and with confidence.

The 5-Phase Java Modernization Workflow

Phase 1 — Analyze (Ask mode)
Read the codebase without touching anything. Use Ask mode + @pom.xml or @build.gradle to understand the current state: Java version, dependencies, deprecated APIs, test coverage.
Phase 2 — Plan (Plan mode)
Generate a detailed upgrade plan. Bob identifies all changes needed, estimates risk per change, and produces a sequenced migration roadmap. Review carefully before proceeding.
Phase 3 — Infrastructure (Code mode)
Update build configuration first: pom.xml Java version, compiler plugin, dependency versions. Build must still compile cleanly after this phase.
Phase 4 — Source code migration (Code mode, incrementally)
File by file or package by package. After each batch: checkpoint → compile → run tests. Never batch too many changes at once. Bob respects this constraint if you include it in your rules.
Phase 5 — Validate & document (Code + Ask modes)
Full test suite pass, Docker build verification, Bob-generated summary of all changes made. Ask Bob to generate a migration ADR (Architecture Decision Record).

What is IBM Bob and how is it different from GitHub Copilot? ▼
IBM Bob is a full SDLC AI partner — it not only completes code but plans, reviews, debugs, writes docs, creates PRs, runs shell commands, and integrates with external tools via MCP. For Enterprise plan customers, IBM commits not to use your prompts or code for model training. For free and standard plans, please review IBM's current privacy policy. Always confirm this in your Enterprise agreement.
Can I choose which AI model Bob uses? ▼
Bob selects the underlying model based on the mode and task. Advanced mode uses more capable (and more expensive) models. Currently, end users cannot manually select a specific model version. Enterprise customers may have additional configuration options — ask your IBM account team.
How do I start a new task cleanly? ▼
Click the "New Conversation" button or type /new in the chat. This clears the context window. For ongoing work, use checkpoints instead so you can roll back without losing everything.
Bob made changes I didn't want. How do I undo them? ▼
Scroll up in the chat to find the last checkpoint before the unwanted changes. Click Restore Checkpoint → Restore Files & Task. This rolls back all files to that state. If no checkpoint exists, use git: git checkout . to discard all unstaged changes, or git stash to stash them.
Why is Bob not responding? ▼
Check these in order: (1) Network connection — Bob requires active internet. (2) IBMid session — you may need to re-authenticate. (3) Bobcoin balance — you may have hit your budget limit. (4) Firewall — corporate proxies may block Bob's backend. See the Connection Troubleshooting section.
Can Bob run commands in my terminal automatically? ▼
Yes, in Code and Advanced modes, Bob can propose and execute shell commands — but only with your approval (unless you've enabled auto-approve for commands, which we don't recommend). Bob will show you the exact command before running it.
What is the .bobignore file and why should I use it? ▼
It's a gitignore-style file that tells Bob which files to never read or modify. You should use it to protect .env files, secrets, compiled artifacts, and any files that would confuse Bob or expose sensitive information. Create it in your project root.
How do I move the Bob panel? ▼
Right-click the Bob panel header and choose "Move Panel" from the context menu. You can dock it left, right, or bottom. Alternatively, drag the panel by its header to reposition it.
What are MCP servers and do I need them? ▼
MCP (Model Context Protocol) servers extend Bob with external integrations like GitHub, databases, or Jira. They are optional — Bob works perfectly without them. Can I use Bob Shell without the IDE? ▼
Yes. Bob Shell is a standalone CLI that works in any terminal — zsh, bash, PowerShell. You don't need VS Code installed. Install Bob Shell via the standard Bob installer and run bob-shell in your terminal.
Can I share custom modes and rules with my team? ▼
Yes. Custom modes are stored in .bob/ directories in your project. Commit these to git and your entire team shares the same modes and rules. This is the recommended way to standardize AI behavior across a team.
What is the difference between a Skill and a Mode? ▼
A Mode is a persistent AI persona with defined tool permissions (what Bob can access: read/write/execute). A Skill is a Markdown document with instructions, procedures, and domain knowledge (what Bob knows and does for a task). Modes control permissions; Skills encode expertise. Use them together for the best results — e.g., activate a Java Modernizer mode and load a Java Upgrade Skill.

Spring Boot 3 requires Java 17+ and renames all javax.* APIs to jakarta.*. Bob handles this systematically: it reads your pom.xml, identifies all affected imports, then renames them package by package. SKILL: Spring Boot 3 Migration

# .bob/rules/spring-boot-3-migration.md

## Skill: Spring Boot 2 → 3 Migration Assistant

### Context
Spring Boot 3 requires Java 17+ and migrates from javax.* to jakarta.* packages.
Spring Security 6 removes WebSecurityConfigurerAdapter.
Spring Data changes: findById returns Optional, some JPA methods renamed.

### Procedure
1. READ @pom.xml first. Confirm current Spring Boot version and Java version.
2. VERIFY Java 17+ is configured as compiler source/target.
3. LIST all javax.* imports across the project using search_files.
4. RENAME javax → jakarta, package by package. Compile after each package.
5. FIND all classes extending WebSecurityConfigurerAdapter.
   REPLACE with SecurityFilterChain @Bean pattern.
6. CHECK Spring Data: CrudRepository.findOne() → findById().orElse(null)
7. RUN full test suite. Fix failures before proceeding.
8. UPDATE any Spring Cloud dependencies to Boot 3 compatible versions.

### Constraints
- NEVER change business logic during migration
- ONE package at a time — compile after each
- CHECKPOINT before steps 3, 5, and 7

### Output format
After each step: ✅ DONE / ⚠️ BLOCKED + reason / 🔴 FAILED + error

## Skill: OWASP Top 10 Security Audit for Java

### Checklist to run systematically

#### A01 - Broken Access Control
- Look for @PreAuthorize / @Secured annotations missing on sensitive endpoints
- Check for IDOR: controller methods that take IDs as parameters
  without validating the current user owns that resource

#### A02 - Cryptographic Failures
- grep for: MD5, SHA1, DES, ECB mode, Random (not SecureRandom)
- Check for hardcoded IVs or encryption keys

- Flag ALL String-concatenated SQL (CRITICAL)

- Check JPQL/HQL for string concatenation

#### A05 - Security Misconfiguration
- Spring Security: is CSRF disabled? Is .permitAll() too broad?

- Debug mode enabled in production profile

#### A06 - Vulnerable Components
- List all dependencies with known CVEs (use search_files on pom.xml)
- Flag: Log4j < 2.17, Spring < 5.3.x, older Jackson versions

#### A08 - Insecure Deserialization
- Flag: ObjectInputStream.readObject() with untrusted data
- Check for Java serialization in REST endpoints
- Look for: XStream, Kryo with default config

- Verify passwords/tokens are never logged
- Check for: log.info("password: " + password)

SKILL: JUnit 5 Test Generation

# .bob/rules/junit5-test-generation.md

## Skill: JUnit 5 Test Generation Standards

### Test naming convention
Method: should_[outcome]_when_[condition]
Example: should_throwNotFoundException_when_userIdDoesNotExist()

### Required coverage per class
- Happy path (normal input, expected output)
- Null inputs for every nullable parameter
- Boundary values (0, -1, Integer.MAX_VALUE, empty string, empty list)
- Exception paths (service throws, repo throws)
- Concurrent access (if the class has state)

### Imports to always include
```java
import org.junit.jupiter.api.*;
import org.mockito.junit.jupiter.MockitoExtension;
import static org.assertj.core.api.Assertions.*;
import static org.mockito.BDDMockito.*;
```

### Mockito pattern (BDD style preferred)
```java
// Arrange
given(userRepository.findById(1L)).willReturn(Optional.of(user));
// Act
User result = userService.getUser(1L);
// Assert
assertThat(result.getName()).isEqualTo("Alice");
```

### Exception assertion pattern
```java
assertThatThrownBy(() -> userService.getUser(999L))
    .isInstanceOf(UserNotFoundException.class)
    .hasMessageContaining("999");
```

### Testcontainers for integration tests
- Use @Testcontainers + @Container for real DB tests
- Never use H2 in-memory for integration tests of JPA queries
- Tag integration tests with @Tag("integration")

### What NOT to do
- Never use @InjectMocks if you can use constructor injection
- Never assert on toString() output
- Never catch exceptions in tests — let JUnit handle them

SKILL: API Design Review

# .bob/rules/api-design-review.md

## Skill: REST API Design Review for Spring Boot

### Checklist
- HTTP verbs: GET (read), POST (create), PUT (replace), PATCH (update), DELETE
- HTTP status codes: 200/201/204 for success, 400/404/409/422 for client errors, 500 for server
- Pagination: use Spring Data Pageable + Page response with metadata
- Error responses: consistent ErrorResponse DTO with code, message, timestamp
- Versioning: /api/v1/... in path or Accept: application/vnd.app.v1+json header
- No verbs in URLs: /api/users/{id}/activate (POST) not /api/activateUser
- Idempotency: PUT and DELETE must be idempotent
- Validation: @Valid on all @RequestBody with proper @NotNull/@Size constraints

### Security checklist for each endpoint
- Is it behind authentication?
- Does it validate ownership (current user can only access their own resources)?
- Is sensitive data masked in responses (no passwords, partial card numbers)?
- Are there rate limits for sensitive endpoints (login, password reset)?

### Output format
For each endpoint found: METHOD /path → STATUS: ✅ OK / ⚠️ ISSUE + fix

// Avant (Java 8 style — beaucoup de boilerplate)
public class Money {
    private final BigDecimal amount;
    private final String currency;
    public Money(BigDecimal amount, String currency) {
        this.amount = amount; this.currency = currency;
    }
    // getters, equals, hashCode, toString... 40 lignes de boilerplate
}

// Après (Java 16+ record — tout généré)
public record Money(BigDecimal amount, String currency) {
    // Compact constructor pour validation
    public Money {
        Objects.requireNonNull(amount, "amount must not be null");
        Objects.requireNonNull(currency, "currency must not be null");
        if (amount.compareTo(BigDecimal.ZERO) < 0)
            throw new IllegalArgumentException("amount must be positive");
    }

    // Méthodes métier sur les records
    public Money add(Money other) {
        if (!this.currency.equals(other.currency))
            throw new IllegalArgumentException("Cannot add different currencies");
        return new Money(this.amount.add(other.amount), this.currency);
    }

    // equals/hashCode/toString générés automatiquement ✅
}

📄Appendice — AGENTS.md : Contrôle des agents en Java enterprise

Concept clé : AGENTS.md (et CLAUDE.md, .bob/rules/) ne sont pas de la documentation — ce sont des surfaces de contrôle architecturales. Ils définissent comment un agent autonome explore votre repository, influençant coût, latence et qualité des résultats.

Le Paradoxe de l'Efficacité — Plus de contexte = pires résultats

Une étude empirique publiée sur arXiv:2602.11988 a testé 3 scénarios sur des repositories réels :

ScénarioTaux de succèsCoût d'inférence
Aucun fichier de contexteBaselineBaseline
Fichier généré par LLMBaseline − 3%+20%
Fichier écrit par un humainBaseline + 4%+coût modéré

Pourquoi ? Un agent lisant un long AGENTS.md avec tous les détails architecturaux essaie d'honorer chaque instruction. Il explore plus de fichiers, lance plus de tests, vérifie plus de contraintes globales — même pour une modification locale mineure. Dans un monorepo Java avec des milliers de classes, c'est catastrophique pour les Bobcoins et la durée des tâches.

Agency Efficiency Principle : La fiabilité autonome ne vient pas d'ajouter plus d'instructions — elle vient de les limiter soigneusement. Objectif : moins de 150 lignes, idéalement moins de 100. Chaque ligne doit répondre à : "Si on retire cette ligne, l'agent fera-t-il une erreur systémique et irrécupérable ?" Si non → la retirer.

✅ Ce qu'il faut mettre dans AGENTS.md pour Java

TEMPLATE AGENTS.md — Java Enterprise (sous 100 lignes)
# AGENTS.md

## Build Strategy
ALWAYS use parallel build: mvn clean install -T 1C -DskipTests
For tests only: mvn test -T 1C
NEVER run mvn clean install without -DskipTests unless explicitly asked to run all tests.
Batch multiple Maven commands into a single execution step.

## Architecture Boundaries
- Layer order: controller → service → repository → entity
- Controllers NEVER access JPA entities directly — map to DTOs via *Mapper interfaces
- Only @Service methods may be annotated @Transactional
- No direct calls from controller to repository layer

## Performance Rules
- NEVER load @OneToMany or @ManyToMany collections in a loop or stream().map()
- Always use JOIN FETCH or @EntityGraph when loading Order with OrderItems
- For financial fields: use BigDecimal with NUMERIC(19,4) in DB. Never float/double.

## Security Non-Negotiables
- No hardcoded passwords, tokens, or API keys — use ${env.SECRET_NAME}
- SQL queries: only @Query with :param binding or JPA methods. No string concatenation.
- File downloads: always validate path stays within /app/uploads/ before serving

## Module Structure (monorepo)
- /api-gateway : Spring Cloud Gateway — routing only, no business logic
- /user-service : user management domain
- /order-service : order lifecycle domain
- /shared-lib    : DTOs and utilities shared across services

## Forbidden Dependencies
- user-service must NOT import order-service classes (use events)
- No direct DB access from api-gateway module

## Test Commands
Unit tests:    mvn test -pl user-service -Dtest="*Test"
Integration:   mvn verify -pl user-service -Dtest="*IT" (requires Docker)
Single test:   mvn test -pl user-service -Dtest="UserServiceTest#should_createUser"

Progressive Disclosure — Monorepos Java

Dans un grand monorepo, un seul AGENTS.md à la racine n'est pas suffisant — mais le remplir de tout est pire. La solution est la progressive disclosure : un fichier racine minimal + des fichiers de contexte dans chaque module.

# Structure recommandée pour un monorepo Java

/AGENTS.md                 ← La carte globale (< 60 lignes)
                              Module structure, build strategy,
                              global non-negotiables

/user-service/AGENTS.md    ← Contraintes spécifiques user-service
                              JPA entities, auth rules, events published

                              Payment state machine, inventory rules

/shared-lib/AGENTS.md      ← Règles de la lib partagée

L'agent lit le fichier le plus proche de son répertoire de travail en premier. Si Bob travaille dans /order-service, il lit /order-service/AGENTS.md puis /AGENTS.md — il n'a jamais besoin des règles de /user-service. Cela maintient le contexte focalisé et économique.

À grande échelle, les fichiers de contexte par repo ne suffisent pas. Les développeurs peuvent les modifier ou les supprimer. La solution : les managed policies — fichiers de contexte déployés au niveau OS sur les machines des développeurs.

  • S'appliquent à tous les repositories sur la machine — impossibles à contourner via config locale.
  • Déployées via MDM/endpoint management (Jamf, Intune, Ansible…).
  • Exemples de règles enterprise : interdiction de credentials hardcodés, obligation d'utiliser le registre Maven interne, règles de licensing, sanitisation des logs.
  • Même règle s'applique : moins de 200 lignes. Une managed policy gonflée dégrade les performances sur chaque interaction Bob de toute la company.

Takeaways stratégiques pour votre équipe Java

#PrincipeAction concrète
1Moins de contexte = plus de fiabilitéAuditer vos fichiers de règles existants. Tout ce qui n'est pas une contrainte architecturale → supprimer.
2Le déterminisme appartient aux outilsFormatting → Spotless/Checkstyle. Pas dans AGENTS.md.
3Les boundaries doivent être explicitesÉcrire les règles de couche (controller → service → repo) dans AGENTS.md. Bob ne peut pas les deviner.
4Les contraintes de performance doivent être encodéesÉcrire la règle N+1 query explicitement. C'est le bug #1 généré par les LLMs sur des projets Hibernate.
5Gouvernance au-dessus du repoPour les règles de sécurité non-négociables (credentials, licensing), utiliser les managed policies, pas AGENTS.md.
6Progressive disclosure dans les monoreposUn AGENTS.md minimal à la racine + fichiers contexte dans chaque module/service.

ARTICLE 03 · Avr 2026 · SkillsJars for Java: Stop Copying AI Skills ↗

SkillsJars — Versionner les Skills comme des dépendances Maven

Le problème : quand on a 5 projets qui copient le même SKILL.md, ils divergent. Un checkout utilise jakarta.ws.rs, un autre garde les anciens patterns, un troisième utilise la mauvaise extension REST. Une fois que l'agent génère des patches qu'on merge réellement, les instructions font partie du pipeline de livraison. Si elles dérivent, l'agent dérive.

La solution de Markus : traiter les Skills comme des JAR Maven — packager, versionner, livrer, consommer comme n'importe quelle lib interne.

Comment SkillsJars fonctionne

📦
1. Écrire les Skills
SKILL.md dans skills/quarkus-rest/, skills/quarkus-native/, etc. Avec front matter YAML : name, description, allowed-tools, license.
🏗️
2. Packager en JAR
Le plugin Maven SkillsJars (com.skillsjars:maven-plugin) scanne skills/ et emballe les SKILL.md dans META-INF/skills/ du JAR.
📤
3. Publier (Maven)
mvn install en local ou déployer sur votre Nexus/Artifactory interne. Versioning sémantique : 1.0.0, 1.1.0-SNAPSHOT.
📥
4. Consommer (goal extract)
Dans le projet consommateur : mvn skillsjars:extract -Ddir=.bob/skills/. Bob lit automatiquement ce dossier.

Le pattern d'intégration LangChain4j avec Bob

Ajouter LangChain4j incrementalement

Bob annote les méthodes de BookingService existant avec @Tool, génère les descriptions en langage naturel, et met à jour la configuration de l'assistant pour que le LLM connaisse ces capacités. Pas de réécriture — augmentation uniquement.
# Prompt Bob pour l'intégration LangChain4j
@src/main/java/com/example/service/BookingService.java
@src/main/java/com/example/entity/

1. Add LangChain4j Quarkus extension for the AI provider we configure
